# Architecture — FieldNotes RAG Lab

## Overview

FieldNotes RAG Lab implements a fully deterministic, offline RAG pipeline in plain Python. No neural network downloads, no API keys, no cloud services. The entire stack runs on CPython with only stdlib dependencies.

---

## Pipeline Stages

### 1. Document Loading (`loader.py`)

Reads `.md` and `.txt` files from a corpus directory. For each file:
- Strips YAML front matter (if present) and extracts key–value metadata
- Extracts inline bold-label metadata (e.g., `**Location:** Ridgeline Creek`)
- Extracts the first heading as the document title, falling back to the filename
- Returns a `Document` dataclass with `doc_id`, `path`, `title`, `content`, and `metadata`

**Design choice:** Regex-only parsing, no external markdown library. This keeps the dependency tree empty and works on Python 3.9+.

### 2. Tokenisation (`tokenizer.py`)

All text processing uses a single tokeniser:
- Lower-case normalisation
- Punctuation removal (preserving hyphens and apostrophes)
- Stop-word filtering (English, ~100 words)
- Support for bigrams via `ngrams()`
- Term-frequency computation (`term_frequency()`)

**Design choice:** No NLTK/spaCy. Field-guide text is domain-specific; the stop-word list is hand-curated to avoid over-stripping species and habitat terms.

### 3. Chunking (`chunker.py`)

Splits documents into overlapping windows using sentence-boundary detection:
- Splits on `.!?` boundaries followed by a capital letter
- Accumulates sentences until the chunk reaches `chunk_size` words
- Advances the cursor by `chunk_size - chunk_overlap` words (sliding window)
- Tracks exact line numbers by scanning character offsets in the original text

Each `Chunk` stores: `chunk_id`, `doc_id`, `source_path`, `title`, `text`, `start_line`, `end_line`, and a copy of document metadata.

**Design choice:** Word count–based chunking (not character count) is more robust to the varied line lengths in Markdown. Sentence-aware splitting avoids mid-sentence breaks that corrupt context.

### 4. Indexing (`index.py`)

Builds a two-component hybrid index:

#### TF-IDF Component
- Computes term frequency (TF) over unigrams + bigrams for each chunk
- Computes smoothed inverse document frequency (IDF) across the corpus
- Stores a L2-normalised TF-IDF vector per chunk

IDF formula: `log((1 + N) / (1 + df)) + 1`

#### Hashed Feature Component
- Hashes each token (unigrams + bigrams) via MD5 into a 1024-bucket sparse vector
- L2-normalises the resulting vector
- Computes cosine similarity at query time

#### Hybrid Score
```
score = alpha * tfidf_dot_product + (1 - alpha) * cosine_hashed
```
Default `alpha = 0.6` gives more weight to lexical exact matches, which matters for species names and technical field terms. The hashed component provides soft-matching for paraphrases.

**Design choice:** MD5-based hashing avoids the dependency on a learned embedding model while still providing a vector-space similarity signal. It is not as powerful as dense embeddings but is fast, deterministic, and entirely offline.

### 5. Retrieval (`retriever.py`)

- Applies query expansion via a hand-coded synonym table (common field-research terms: stream ↔ creek/riffle/river, bird ↔ avian/passerine, etc.)
- Scores all indexed chunks against the expanded query
- Filters by a minimum score threshold
- Deduplicates by source document (max 2 chunks per document by default)
- Returns top-k `(IndexedChunk, score)` pairs

### 6. Citation Model (`citation.py`)

Converts ranked `(chunk, score)` pairs into `Citation` objects with:
- Sequential reference numbers `[1]`, `[2]`, …
- Source path and line range
- Relevance score
- 25-word snippet of the chunk text

Citations can be formatted as inline markers, bibliography entries, or one-line short forms.

### 7. Answer Synthesis (`synthesis.py`)

Constructs an answer by:
1. Extracting the first N sentences from each top-k chunk
2. Appending the inline citation marker `[ref_num]` to each passage
3. Joining passages with a space
4. Appending a bibliography section

This approach is fully deterministic — the same query + index always produces the same answer. The answer is grounded in retrieved text, not generated by a language model, which makes it auditable and verifiable.

### 8. Evaluation (`evaluator.py`)

Runs a fixture-driven evaluation with four metrics:
- **Recall@k** — fraction of expected documents that appear in top-k results
- **MRR** — mean reciprocal rank of the first expected document
- **Citation accuracy** — whether the answer contains at least `min_citations` citations
- **Answer coverage** — fraction of expected keywords present in the answer

Composite score: `0.35 × recall + 0.25 × mrr + 0.20 × citation_acc + 0.20 × coverage`

## Data Flow Diagram

```
Corpus dir (.md/.txt)
        │
        ▼
   DocumentLoader
        │ Document(doc_id, title, content, metadata)
        ▼
      Chunker
        │ [Chunk(chunk_id, text, start_line, end_line, ...)]
        ▼
   HybridIndex.build()
        │ IDF map, TF-IDF vectors, hashed feature vectors → saved to JSON
        ▼
   HybridIndex.score(query)
        │ [(IndexedChunk, score), ...]
        ▼
     Retriever
        │ expanded query → deduped top-k results + Citations
        ▼
    Synthesizer
        │ answer text with [1], [2]… markers
        ▼
  SynthesisResult(query, answer, citations, num_chunks_used)
```

---

## Persistence

The index is serialised to a single JSON file (default: `.fieldnotes-index.json`). This file contains:
- Configuration values (alpha, n_buckets)
- IDF vocabulary map
- Per-chunk TF-IDF vectors (keyed by term string)
- Per-chunk hashed feature vectors (keyed by bucket number as string)

The file can be inspected with any JSON viewer and shared as a plain file.

## Scalability Notes

| Corpus size | Expected index size | Index build time | Query time |
|---|---|---|---|
| 10 documents, ~5000 words | ~200 KB JSON | < 0.1s | < 1ms |
| 100 documents, ~50 000 words | ~2 MB JSON | < 1s | < 5ms |
| 1000 documents, ~500 000 words | ~20 MB JSON | < 10s | < 50ms |

For very large corpora (>1M words), consider sharding the corpus into thematic sub-indices and routing queries to the relevant shard before scoring.
